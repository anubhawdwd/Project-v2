# SECTION 1 — PYTHON FUNDAMENTALS

# 1. Mutable Default Arguments

## Interview Question

Why are mutable default arguments dangerous in Python?

## Strong Interview Answer

Default arguments in Python are evaluated only once at function definition time, not every time the function is called.

So if a mutable object like a list or dictionary is used as a default parameter, the same object instance gets reused across function calls.

Example:

```python

def append_item(item, items=[]):
    items.append(item)
    return items
```

Calling:

```python
append_item(1)
append_item(2)
```

returns:

```python
[1]
[1, 2]
```

because the same list object is reused.

Correct approach:

```python

def append_item(item, items=None):
    if items is None:
        items = []

    items.append(item)
    return items
```

---

## Cross Questions

### Q1. Why does Python behave this way?

Because evaluating default arguments once improves performance and keeps function definitions consistent.

---

### Q2. Is this behavior ever useful?

Yes.
It can intentionally preserve state between calls.

Example:

```python

def counter(count=[0]):
    count[0] += 1
    return count[0]
```

But this is rarely recommended in production code because it reduces readability.

---

### Q3. What production bugs can this cause?

* State leakage between requests
* Shared cache corruption
* Unexpected API behavior
* Memory retention issues

Especially dangerous in:

* FastAPI apps
* Celery workers
* Long-running services

---

## Common Mistake

Saying only:

> “Mutable defaults are bad.”

Senior interviews expect reasoning about object lifetime and evaluation timing.

# 2. Shallow Copy vs Deep Copy

## Interview Question

Explain shallow copy vs deep copy.

## Strong Interview Answer

A shallow copy creates a new outer container but still references nested objects.

A deep copy recursively copies all nested objects.

Example:

```python
import copy

original = [[1, 2], [3, 4]]

shallow = copy.copy(original)
deep = copy.deepcopy(original)
```

If we modify:

```python
original[0].append(99)
```

Then:

```python
shallow -> affected
deep -> unaffected
```

because shallow copy shares inner references.

---

## Cross Questions

### Q1. Why does shallow copy exist?

Deep copy is expensive.
Shallow copy is faster and often sufficient.

---

### Q2. Where do deep copies become dangerous?

* Large nested objects
* Huge configuration trees
* Data pipelines
* High-memory systems

Deep copying large objects can severely impact performance.

---

### Q3. How do ORMs relate to this?

ORM objects often contain nested relationships.
Accidental copying can duplicate references incorrectly or cause stale state problems.

# 3. Python Memory Management

## Interview Question

How does memory management work in Python?

## Strong Interview Answer

Python primarily uses:

1. Reference counting
2. Cyclic garbage collection

Every object maintains a reference count.
When the reference count reaches zero, memory can be reclaimed.

Problem:
Reference counting alone cannot handle cyclic references.

Example:

```python
class A:
    pass

obj1 = A()
obj2 = A()

obj1.ref = obj2
obj2.ref = obj1
```

Both objects reference each other.
Their reference counts never become zero.

That is why Python also has cyclic garbage collection.

---

## Cross Questions

### Q1. Why do long-running workers slowly consume more memory?

Possible reasons:

* Reference leaks
* Large retained objects
* Fragmentation
* Global caches
* Unclosed connections
* Batch accumulation

---

### Q2. How would you debug memory leaks?

Tools:

* tracemalloc
* objgraph
* memory_profiler
* gc module

Production strategy:

* Compare snapshots
* Inspect retained object graphs
* Monitor heap growth
* Recycle workers when necessary

---

### Q3. What is memory fragmentation?

Even after objects are deleted, memory may not return efficiently to the OS because allocations become fragmented internally.

# 4. Generators and Lazy Evaluation

## Interview Question

Why are generators important in backend systems?

## Strong Interview Answer

Generators enable lazy evaluation.

They produce values one at a time instead of loading everything into memory.

Example:

```python

def read_large_file(path):
    with open(path) as f:
        for line in f:
            yield line
```

Benefits:

* O(1) memory usage
* Streaming support
* Better scalability
* Efficient processing pipelines

---

## Cross Questions

### Q1. What is generator exhaustion?

A generator can only be consumed once.
After iteration completes, it becomes exhausted.

---

### Q2. Difference between iterator and generator?

Iterator:
Any object implementing:

```python
__iter__()
__next__()
```

Generator:
A simpler way to create iterators using `yield`.

---

### Q3. Where are generators useful in production?

* Streaming APIs
* Kafka consumers
* CSV exports
* ETL jobs
* Log processing
* Large DB iteration

---

# 5. Decorators

## Interview Question

Explain decorators in Python.

## Strong Interview Answer

Decorators wrap functions to extend behavior without modifying the original implementation.

Common production uses:

* Logging
* Authentication
* Retries
* Metrics
* Tracing
* Caching

Example:

```python
from functools import wraps


def logger(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        print("Calling function")
        return func(*args, **kwargs)
    return wrapper
```

---

## Cross Questions

### Q1. Why use functools.wraps?

Without it, metadata like:

* function name
* docstring
* annotations

gets lost.

---

### Q2. What is exponential backoff?

Retry delay increases exponentially after failures.

Example:

```python
1s -> 2s -> 4s -> 8s
```

This reduces retry storms.

---

### Q3. What is jitter?

Randomized retry delays.
Used to prevent synchronized retries from multiple systems.

# SECTION 2 — CONCURRENCY & ASYNC

# 6. The GIL

## Interview Question

What is the GIL?

## Strong Interview Answer

The Global Interpreter Lock allows only one thread to execute Python bytecode at a time within a process.

This means Python threads are not truly parallel for CPU-bound work.

---

## Cross Questions

### Q1. When do threads still help?

Threads work well for I/O-bound tasks because threads release the GIL during blocking operations like:

* network calls
* file I/O
* DB operations

---

### Q2. Best approach for CPU-heavy tasks?

Multiprocessing.
Because separate processes have separate Python interpreters and separate GILs.

---

### Q3. Why can threading reduce performance?

* Context switching overhead
* GIL contention
* Increased synchronization cost

---

## Production Perspective

Use:

| Scenario              | Recommended       |
| --------------------- | ----------------- |
| API calls             | asyncio/threading |
| Image processing      | multiprocessing   |
| ML inference          | multiprocessing   |
| High concurrency APIs | asyncio           |

---

# 7. Asyncio and Event Loops

## Interview Question

How does asyncio work?

## Strong Interview Answer

Asyncio uses cooperative multitasking.

Tasks voluntarily yield control during await operations.

The event loop schedules tasks efficiently while waiting for I/O.

---

## Cross Questions

### Q1. What blocks the event loop?

* requests library
* time.sleep()
* CPU-heavy loops
* synchronous DB drivers

inside async functions.

---

### Q2. Why is blocking dangerous?

One blocked event loop slows every request handled by that worker.

---

### Q3. How do you perform async HTTP requests?

Using async clients like:

* httpx
* aiohttp

---

### Q4. Should everything become async?

No.
Async adds complexity.
Only use it when concurrency benefits outweigh maintenance cost.

---

# 8. Threading vs Multiprocessing vs Asyncio

## Interview Question

Compare threading, multiprocessing, and asyncio.

## Strong Interview Answer

| Technique       | Best For            | Limitation                 |
| --------------- | ------------------- | -------------------------- |
| Threading       | I/O-bound tasks     | GIL                        |
| Multiprocessing | CPU-bound tasks     | Higher memory usage        |
| Asyncio         | High concurrent I/O | Requires non-blocking code |

---

## Cross Questions

### Q1. What is IPC?

Inter-process communication.
Processes communicate via:

* pipes
* queues
* sockets
* shared memory

---

### Q2. Why is multiprocessing expensive?

* Process creation cost
* Separate memory space
* Serialization overhead

# SECTION 3 — DATABASES

# 9. Indexes in PostgreSQL

## Interview Question

How do indexes improve performance?

## Strong Interview Answer

Indexes reduce lookup time by avoiding full table scans.

Without indexes:

```sql
SELECT * FROM users WHERE email='abc@test.com';
```

requires scanning every row.

With indexes:
Database can directly locate matching rows.

---

## Cross Questions

### Q1. Why not index everything?

Indexes increase:

* write cost
* storage usage
* update overhead

Too many indexes can slow inserts and updates.

---

### Q2. What is a composite index?

An index across multiple columns.

Example:

```sql
CREATE INDEX idx_user_status
ON users(country, status);
```

---

### Q3. What is the leftmost prefix rule?

Composite indexes work efficiently only from the leftmost indexed columns.

---

# 10. Transactions and Isolation Levels

## Interview Question

What is a transaction?

## Strong Interview Answer

A transaction is a sequence of operations executed as a single atomic unit.

ACID properties:

* Atomicity
* Consistency
* Isolation
* Durability

---

## Cross Questions

### Q1. What is dirty read?

Reading uncommitted data from another transaction.

---

### Q2. What is a deadlock?

Two transactions waiting indefinitely for each other’s locks.

---

### Q3. How do you reduce deadlocks?

* Consistent lock ordering
* Smaller transactions
* Proper indexing
* Reduced lock duration

---

# 11. N+1 Query Problem

## Interview Question

What is the N+1 problem?

## Strong Interview Answer

Instead of one optimized query, the application executes many smaller queries.

Example:

1 query for users.
Then N separate queries for each user’s orders.

This creates severe DB overhead.

---

## Cross Questions

### Q1. How do ORMs solve this?

Using:

* eager loading
* joins
* select_related
* prefetch_related

---

### Q2. Why is N+1 dangerous?

* Increased latency
* DB overload
* Connection exhaustion

# SECTION 4 — REDIS & CACHING

# 12. Redis Caching

## Interview Question

What caching strategies do you know?

## Strong Interview Answer

Common strategies:

| Strategy      | Description                 |
| ------------- | --------------------------- |
| Cache Aside   | App checks cache first      |
| Write Through | Cache updated immediately   |
| Write Back    | Delayed DB writes           |
| Read Through  | Cache fetches automatically |

Most backend systems commonly use cache-aside.

---

## Cross Questions

### Q1. What is cache stampede?

Many requests simultaneously miss cache and overload the DB.

---

### Q2. How do you prevent cache stampede?

* TTL randomization
* Distributed locks
* Request coalescing
* Background refresh

---

### Q3. Why should TTLs sometimes be randomized?

To avoid simultaneous expiration spikes.

---

# 13. Distributed Rate Limiting

## Interview Question

How would you implement distributed rate limiting?

## Strong Interview Answer

Redis is commonly used because atomic operations are fast.

Typical flow:

* Increment request count
* Set expiration window
* Reject requests exceeding threshold

---

## Cross Questions

### Q1. Why use Redis INCR?

INCR is atomic.
It prevents race conditions.

---

### Q2. What algorithms exist?

* Fixed Window
* Sliding Window
* Token Bucket
* Leaky Bucket

---

### Q3. Which is most accurate?

Sliding window.
But it is more computationally expensive.

# SECTION 6 — CELERY & BACKGROUND JOBS

# 15. Celery Fundamentals

## Interview Question

Why use Celery?

## Strong Interview Answer

Celery enables asynchronous background task execution.

Useful for:

* emails
* notifications
* report generation
* image processing
* scheduled jobs

This prevents long-running operations from blocking APIs.

---

## Cross Questions

### Q1. What is idempotency?

An operation produces the same result even if executed multiple times.

Critical because distributed systems can retry tasks.

---

### Q2. Why can tasks execute multiple times?

* worker crashes
* broker reconnects
* acknowledgment failures
* retries

---

### Q3. What is a dead letter queue?

A queue storing repeatedly failed tasks for later investigation.

## Interview Question

## Strong Interview Answer

2. Identify retained objects
3. Check global caches
4. Inspect batch accumulation
5. Analyze reference cycles
6. Monitor task payload sizes
7. Recycle workers if needed

Tools:

* tracemalloc
* objgraph
* gc
* memory_profiler
